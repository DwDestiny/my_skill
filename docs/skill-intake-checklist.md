# Skill 入库验收清单

## 候选确认

在创建或更新 Skill 前，必须先向用户确认：

- 适用场景：什么时候应该触发这个 Skill。
- 解决问题：它解决哪个稳定、可复用的问题。
- 预期收益：节省时间、降低错误、统一口径，还是提升产出质量。
- 计划动作：新建还是更新。
- 目标路径：准备落到哪个 `skills/<skill_slug>/`。

未获得明确确认前，不落盘。

## 入库检查

- 目录位于 `skills/<skill_slug>/`。
- `SKILL.md` 存在，且 frontmatter 包含 `name` 和 `description`。
- `description` 写清触发场景，不只写能力名称。
- `SKILL.md` 正文聚焦核心工作流，避免把参考资料全塞进去。
- 长资料放在 `references/`，并在 `SKILL.md` 中说明什么时候读取。
- 重复且易错的操作优先脚本化，放在 `scripts/`。
- README 的 Skill 目录已同步更新。
- 不包含密钥、账号、令牌、私有日志。

## 交付检查

- 本地 Git 状态干净，或清楚列出未提交原因。
- 关联 GitHub issue。
- 提交信息说明本次新增或更新的 Skill。
- 若 Skill 包含脚本，已完成最小运行验证。
- 若 Skill 包含外部依赖，已在文档中写清依赖边界和降级方式。
