## 1. JTBD (Jobs-to-be-Done)

### 核心概念
用户不是购买产品，而是"雇佣"产品来完成某个特定的"工作"。JTBD关注用户想要取得的进步，而非产品功能。

### 三个层次的任务

| 层次 | 说明 | 提问方式 |
|------|------|----------|
| 功能性任务 | 实际要完成的事情 | "用户用这个要完成什么具体的事？" |
| 情感任务-个人 | 用户想感受到什么 | "使用这个让用户感觉如何？" |
| 情感任务-社会 | 用户想展示什么 | "用户使用这个希望别人怎么看他？" |

### 推进力量 (Forces of Progress)

推动用户改变的力量：
- **推动力量**: 当前方案的不满 (push of the situation)
- **拉动力量**: 新方案的吸引力 (pull of the new solution)
- **焦虑力量**: 对新方案的担忧 (anxiety of the new solution)
- **惯性力量**: 现状的舒适感 (habit of the present)

**实践**: 帮助用户分析这四个力量，理解用户的真实决策过程。

## 2. Design Thinking

### 五阶段流程

1. **共情 (Empathize)**: 深入理解用户需求和痛点
2. **定义 (Define)**: 明确核心问题和洞察
3. **构思 (Ideate)**: 头脑风暴产生多种解决方案
4. **原型 (Prototype)**: 快速构建低成本原型
5. **测试 (Test)**: 验证假设并收集反馈

### 与Skill的映射

| Design Thinking阶段 | Skill对应Phase |
|-------------------|---------------|
| 共情 + 定义 | Phase 0 出发探知 |
| 构思 | Phase 1 想法完善 |
| 原型 + 测试 | Phase 3 MVP规划 |

## 3. Lean Startup & MVP

### Build-Measure-Learn 循环

```
想法(Idea) → 构建(Build) → 产品(MVP) → 衡量(Measure) → 数据(Data) → 学习(Learn) → 新的想法
```

### MVP的三种类型

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| 视频型MVP | 用视频展示产品概念，看用户是否感兴趣 | 需要验证需求是否存在 |
| 着陆页MVP | 用简单页面收集用户意向和邮箱 | 需要验证市场兴趣 |
| 手动服务MVP | 用人工手动完成产品功能，用户以为是自动的 | 需要验证核心流程 |
| 单功能MVP | 只开发最核心的一个功能 | 需要验证解决方案是否有效 |

### 创新核算 (Innovation Accounting)

MVP阶段的指标：
- **可执行指标**: 用户转化率、留存率、付费率（能指导行动的）
- **虚荣指标**: 总下载量、总用户数（看起来好看但不能指导行动）

## 4. 苏格拉底式提问法

### 核心原则
不直接给答案，通过连续追问引导对方自己发现真相。

### 六种问题类型

| 类型 | 目的 | 示例 |
|------|------|------|
| 澄清问题 | 确保理解准确 | "你说的X具体是指什么？" |
| 假设探查 | 揭示隐含假设 | "你这个想法基于什么假设？" |
| 证据追问 | 检验依据 | "你为什么这么认为？有数据支撑吗？" |
| 视角转换 | 换个角度思考 | "如果你是你说的用户，你会怎么想？" |
| 影响推演 | 预判后果 | "如果这样做，最可能发生什么？" |
| 元问题 | 反思问题本身 | "这个问题是最重要的问题吗？" |

### 产品发现中的核心问题清单

1. **Who Exactly Is This For?**
   - 不好: "SMB客户"
   - 好: "SaaS公司10-50人团队的运营经理，目前用Excel管理发票"

2. **What Problem Does This Solve?**
   - 追问: "如果不解决这个问题，会发生什么？"

3. **What Are We Assuming?**
   - 常见危险假设:
     - "用户会每天看这个仪表盘" (他们不会)
     - "用户理解我们用的术语" (他们不理解)
     - "用户会在使用前完成配置" (他们会跳过)

4. **How Will We Know This Succeeded?**
   - 不好: "用户会喜欢它"
   - 好: "40%的活跃用户会在上线30天内使用这个功能"

5. **What's The Smallest Test?**
   - 追问: "验证这个假设的最小实验是什么？"

## 5. 5 Whys 根因分析

### 操作流程
遇到问题时，连续问5次"为什么"，直到找到根本原因。

### 示例

```
问题: 机器停了
  Why1: 为什么停了？→ 保险丝烧了
  Why2: 为什么烧了？→ 轴承润滑不足
  Why3: 为什么不足？→ 润滑泵没打上去
  Why4: 为什么没打？→ 润滑泵轴磨损了
  Why5: 为什么磨损？→ 滤网没清洗，杂质进去了

根本原因: 滤网维护流程缺失
解决方案: 建立滤网定期清洗流程
```

### 在产品中的应用

```
用户说"我想要更快的支付方式"
  Why1: 为什么需要更快？→ 现在支付步骤太多
  Why2: 为什么步骤多是个问题？→ 每次付款要2-3分钟，着急时很烦
  Why3: 为什么着急？→ 主要在赶时间的场景，如食堂排队
  
真实需求: 在特定场景下极速完成小额支付
更优方案: 场景化快捷支付（而非做一个新支付方式）
```

## 6. 商业模式画布

### 九大模块详解

```
┌─────────────────────────────────────────────────┐
│  关键合作         关键业务         价值主张        │
│  (KP)           (KA)            (VP)            │
│                 核心资源                          │
│                 (KR)                              │
├─────────────────────────────────────────────────┤
│  成本结构          │          收入来源            │
│  (CS)             │          (RS)               │
└─────────────────────────────────────────────────┘
         │           │            │
    ┌────┘    ┌──────┘     ┌────┘
    ▼         ▼            ▼
 客户细分   客户关系    渠道通路
   (CS)     (CR)       (CH)
```

| 模块 | 核心问题 | 填写指引 |
|------|---------|----------|
| 客户细分 | 我们为谁创造价值？ | 越具体越好，描述一个具体的人 |
| 价值主张 | 我们为客户解决什么问题？ | 用客户语言描述，避免技术术语 |
| 渠道通路 | 如何触达客户？ | 考虑获客成本和效率 |
| 客户关系 | 与客户建立什么关系？ | 自助服务/社区/专属顾问 |
| 收入来源 | 客户为什么付费？付多少？ | 列出所有可能的收入方式 |
| 核心资源 | 需要什么关键资源？ | 人力/技术/品牌/资金 |
| 关键业务 | 必须做什么关键事情？ | 为了交付价值必须做的事 |
| 关键合作 | 需要哪些合作伙伴？ | 供应商/渠道/技术合作 |
| 成本结构 | 主要成本是什么？ | 固定成本 vs 变动成本 |

### 填写顺序建议

1. 先填右侧（客户细分 → 价值主张）
2. 再填左侧（关键业务 → 核心资源 → 关键合作）
3. 最后填底部（成本结构 → 收入来源）
4. 回到中间确认价值主张与客户细分的匹配

## 8. Kano模型

### 五类需求

| 类型 | 用户反应 | 优先级 | 示例 |
|------|---------|--------|------|
| 基本需求(M) | 有：无所谓 / 没有：非常不满 | **必须做** | 手机能打电话 |
| 期望需求(O) | 有：满意 / 没有：不满 | **越多越好** | 手机续航长 |
| 魅力需求(A) | 有：惊喜 / 没有：无所谓 | **差异化武器** | 手机AI摄影 |
| 无差异需求(I) | 有无都无所谓 | **不做** | 闹钟20种铃声 |
| 反向需求(R) | 有：不满 / 没有：无所谓 | **主动去掉** | 强制广告 |

### Kano双问卷法

对每个功能问两个问题：

| 正向问题 | 反向问题 |
|---------|---------|
| 如果有这个功能，你感觉如何？ | 如果没有这个功能，你感觉如何？ |

**选项**: 1-我很喜欢 2-理应如此 3-无所谓 4-可忍受 5-我不喜欢

### 分类对照表

```
正向\反向 | 很喜欢 | 理应如此 | 无所谓 | 可忍受 | 不喜欢
----------|--------|----------|--------|--------|--------
很喜欢    | Q      | A        | A      | A      | O
理应如此  | R      | I        | I      | I      | M
无所谓    | R      | I        | I      | I      | M
可忍受    | R      | I        | I      | I      | M
不喜欢    | R      | R        | R      | R      | Q

Q = 可疑(回答矛盾)
A = 魅力需求
O = 期望需求
M = 基本需求
I = 无差异需求
R = 反向需求
```

### Better-Worse 敏感度公式

```
满意度影响 (SI) = (A + O) / (A + O + M + I)
不满意度影响 (DSI) = (O + M) / (A + O + M + I) × (-1)

SI 和 DSI 的绝对值越大，该功能对用户"心情波动"的影响越强，优化潜力越高。
```

## 9. MoSCoW方法

### 四个优先级

| 优先级 | 含义 | 选择标准 | MVP中的位置 |
|--------|------|----------|-------------|
| Must Have (M) | 必须有 | 没有它产品不可用 | **MVP核心** |
| Should Have (S) | 应该有 | 重要但可替代 | V1.0 |
| Could Have (C) | 可以做 | 锦上添花 | V2.0+ |
| Won't Have (W) | 不做 | 资源不匹配 | 排除 |

### Must Have的判断标准

一个功能是Must Have当且仅当：
1. 没有它，产品就无法完成核心价值交付
2. 没有它，目标用户无法完成核心任务
3. 缺少它会导致用户放弃使用
4. 法律/合规要求必须包含

### MoSCoW的常见误区

- **误区1**: "所有功能都重要" → 如果所有都是Must，等于没有优先级
- **误区2**: Must太多 → Must应该不超过总功能的30%
- **误区3**: Won't意味着永远不 → Won't是当前不做，后续可重新评估

## 10. RICE评分模型

### 公式

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

### 四个维度

| 维度 | 定义 | 评分方式 |
|------|------|----------|
| Reach (影响人数) | 这个功能会影响多少用户？ | 用具体数字（如"每季度5000用户"），然后映射到1-10 |
| Impact (影响力) | 对每个用户的影响有多大？ | 0.5=微小 1=小 2=中 3=大 |
| Confidence (信心) | 对估计有多大信心？ | 用百分比，然后映射到1-10 |
| Effort (工作量) | 需要多少资源？ | 用人天/人月，然后映射到1-10 |

### Reach映射表 (示例)

```
影响用户规模        分数
─────────────────────
>100万/季度        10
50-100万/季度       8
10-50万/季度        6
1-10万/季度         4
<1万/季度           2
无法量化            1
```

### Confidence映射

```
信心来源                分数
─────────────────────
有用户数据支撑           10
有竞品验证               8
有用户反馈               6
团队经验判断             4
纯假设                   2
```

### RICE使用建议

1. RICE不是绝对分数，而是用于**相对比较**
2. 信心分数是保险阀——对不确定的高分项目要降低信心
3. Effort应该用工程师的实际评估，而非PM的猜测
4. 定期重新评分（建议每迭代一次）

## 11. 竞品分析框架

### 竞品识别方法

| 方法 | 说明 |
|------|------|
| 直接竞品 | 解决同样问题、面向同样用户的竞品 |
| 间接竞品 | 解决同样问题、方式不同的竞品 |
| 替代方案 | 用户现在用来解决这个问题的非产品方式 |
| 潜在进入者 | 有能力进入这个市场的公司 |

### 竞品分析维度

```
分析维度              说明
─────────────────────────────────────
目标用户              竞品面向谁？与我们重叠度？

用户体验              竞品的交互设计如何？
价格策略              免费/付费/订阅？价格点？
技术架构              有什么技术优势或限制？
市场定位              竞品如何描述自己？
用户口碑              用户评价如何？痛点在哪？
团队背景              创始团队/投资方/资源
增长速度              用户增长/收入增长数据
```

## 12. 用户故事地图

### 构建步骤

1. **确定用户旅程 backbone**: 从用户视角描述完成目标的主要步骤
2. **在每个步骤下列出活动**: 拆分为更具体的操作
3. **在每个活动下列出故事/功能**: 具体的实现需求

### 示例

```
用户目标: 在线购买一件商品

Backbone:  浏览商品    →   选择商品    →   下单支付    →   确认收货
           ├─ 搜索      ├─ 查看详情    ├─ 填写地址    ├─ 查看物流
           ├─ 分类浏览   ├─ 对比        ├─ 选择支付方式 ├─ 确认收货
           ├─ 推荐       ├─ 加入购物车  ├─ 使用优惠券   ├─ 评价
           └─ 收藏       └─ 查看库存   └─ 确认订单    └─ 售后
```

### 使用用户故事地图确定MVP

在地图上画一条"MVP线"：
- 线以上的功能 = MVP必须包含
- 线以下的功能 = 后续版本

## 13. 情景规划

### 三种情景

| 情景 | 假设 | 规划目的 |
|------|------|----------|
| 乐观情景 | 一切顺利 | 如何抓住机遇加速 |
| 最可能情景 | 正常发展 | 标准执行计划 |
| 悲观情景 | 遇到挫折 | 如何止损和Pivot |

### 情景规划模板

```
情景名称: [乐观/最可能/悲观]

触发条件:
- [什么情况下会进入这个情景]

关键指标:
- [观察什么指标来判断是否进入此情景]
- 阈值: [具体数值]

3个月后的状态:
- 用户量: [范围]
- 收入: [范围]
- 产品状态: [描述]

应对策略:
- [具体行动1]
- [具体行动2]
- [具体行动3]

Pivot信号:
- [如果出现X，需要考虑Pivot]
```

### Pivot类型参考

| Pivot类型 | 说明 | 何时考虑 |
|-----------|------|----------|
| 缩小Zoom-in | 原产品的某个功能成为全部 | 某个功能使用率远超其他 |
| 放大Zoom-out | 原产品成为更大平台的一部分 | 用户需要更多相关功能 |
| 客户细分 | 保持方案，换一批用户 | 当前用户不付费但其他群体可能 |
| 客户需求 | 保持用户，换个问题 | 用户有其他更痛的痛点 |
| 平台 | 从应用变为平台 | 多个应用有共同需求 |
| 技术 | 用新技术解决同一问题 | 新技术大幅降低成本 |
